## 1. 输出语言规则

- 所有对话回复、思考过程中的计划步骤、代码解释、注释内容，必须使用**简体中文**；
- 若涉及技术术语，需补充中文释义（如：“BEM 命名法（块-元素-修饰符，一种 CSS 命名规范）”）

## 2.样式规范

- 优先使用 Tailwind CSS 进行样式开发，确保样式一致性、可维护性与开发效率。避免冗余自定义样式，除非 Tailwind 无法满足特定设计需求。

## 3.代码规范

- 应处理 JavaScript 运行时的边界值，优先使用可选链操作符（?.）进行安全访问，确保在任何数据状态下代码均不抛出异常、不导致白屏
- 所有代码必须兼容主流浏览器（Chrome、Edge、Safari、Firefox）及其近三个稳定版本。避免使用实验性或未广泛支持的语法与 API，确保生产环境稳定运行。
- 避免使用浏览器不支持的最新 CSS 特性，必要时需使用 polyfill 或 fallback 方案。
- 三元运算符嵌套层数不应超过 2 层，避免代码可读性下降。
- 简化代码逻辑，消除过度嵌套：避免 “回调地狱”、多层 if-else/for 嵌套（可通过 Promise/async-await、switch-case、策略模式优化），让代码逻辑平铺直叙，一眼就能看清执行流程
- 消除 “魔幻代码”：移除硬编码（如将 if (status === 1)改为 if (status === StatusEnum.SUCCESS)，用常量 / 枚举替代魔法值），避免模糊不清的变量 / 函数，杜绝无意义的复杂逻辑
- 消除冗余与死代码：删除重复代码（可封装为公共函数 / 组件复用）、删除未被调用的死代码（如废弃的函数、注释掉的代码）、清理无用的临时变量
- 合理抽象封装：将高频复用的逻辑（如接口请求、数据格式化、表单校验）封装为公共工具函数、hooks、类或组件，避免重复编写，后续修改只需改动一处

## 4. 文件与模块组织

- 每个功能模块或组件，应独立封装为一个单独的文件（如：`Button.tsx`、`Modal.js`），避免将所有代码写在一个文件中。
- 目录结构应清晰、合理，避免过深或过宽的层级嵌套，保持代码组织的可维护性。
- 遵循 “高内聚、低耦合” 原则 —— 一个模块 / 组件只负责单一功能，减少模块间的依赖和关联
- 若指令为优化代码，一定要考虑模块化 / 组件化拆分
